## 11. MemoryChannel：节点内低延迟路径

前面介绍三种 Channel 的统一抽象时，我们已经看到，MemoryChannel、PortChannel 和 SwitchChannel 虽然对上提供相似的通信接口，底层却对应完全不同的硬件路径。MemoryChannel 面向的是节点内低延迟通信：当一张 GPU 的显存已经通过 CUDA IPC、VMM 或其他 P2P 机制映射到另一张 GPU 的地址空间后，执行 Kernel 的 GPU 线程就可以像访问普通 Global Memory 一样，直接对远端显存发起 load 和 store。整个过程中不需要 CPU 参与每一次传输，也不需要先把数据复制到通信库的中转 Buffer，因此它是一条典型的 GPU-driven、one-sided、zero-copy 数据路径。

从设备端句柄来看，MemoryChannel 最核心的是本地源地址、远端映射地址和同步对象。以 Kernel 在 GPU 0 上执行为例，src_ 通常指向 GPU 0 的本地 Buffer，dst_ 则是 GPU 1 显存映射到 GPU 0 地址空间后的远端视图。put 会组织多个 GPU 线程从本地 src_ 读取数据，再写入远端 dst_；get 的方向正好相反，它从远端 dst_ 读取，再写回本地 src_。read 和 write 则提供更细粒度的单元素访问，适合读取状态字段、控制信息或少量数据，但不适合让单个线程循环承担大块数据搬运。

普通 put 的真实路径可以概括为“本地 HBM、当前 GPU 的 SM 寄存器、节点内互联、远端 HBM”。参与线程先执行 Global Load，把自己负责的数据片段读入当前线程的寄存器，然后执行面向远端映射地址的 Global Store，请求经过 NVLink、NVSwitch 或 PCIe P2P，最终落到目标 GPU 的显存。get 同样由当前 GPU 的 SM 发起，只是先通过互联从远端 HBM 完成 Global Load，再把返回到线程寄存器的数据写入本地 HBM。这里没有代码层面显式调用 cudaMemcpy，也不是单独提交一条 DMA 任务；真正产生请求的是执行 Kernel 的 GPU 线程。

MemoryChannel 能够获得较高吞吐，并不是因为某一个线程搬得特别快，而是因为大量线程共同生成连续、可合并的内存请求。copy 模板通常让线程按照 threadId、threadId 加 numThreads 的方式分工，每个线程处理若干个 4B、8B 或 16B 元素。连续线程访问连续地址时，一个 Warp 的请求可以被硬件合并为较少的内存事务；多个活跃 Warp 和多个 SM 再进一步堆高 Outstanding 请求数量，用并发隐藏远端访问延迟。因此，地址对齐、访问连续性、线程数量和互联拓扑都会直接影响 put 和 get 的实际带宽。

除普通 Payload 搬运之外，MemoryChannel 还提供 putPackets 和 unpackPackets，用于更强调小消息时延的 Packet 路径。LL16Packet 用 16B 存储 8B Payload，布局可以理解为 data1、flag1、data2、flag2；LL8Packet 用 8B 存储 4B Payload，布局是 data 和 flag。发送端线程从本地数据区读取 Payload，在寄存器中与当前通信轮次的 Flag 组合，再通过 volatile Global Store 直接写入接收端的 Packet Buffer。接收端则使用 volatile Global Load 持续轮询，只有观察到预期 Flag 后才提取 Payload 并写回本地普通数据区。

Packet 的价值在于把数据和到达标志放进同一次远端写入协议中。接收端可以提前启动，某个 Packet 一旦到达就立即消费，不必为每个细小分片单独执行一次显式 signal 和 wait。LL16 中重复放置两个 Flag，还能够在底层至少具备 8B 写入原子性的假设下，帮助识别 16B Packet 只更新了一半的撕裂状态。不过，这条路径并不是没有代价：LL16 和 LL8 的有效 Payload 比例都只有百分之五十，接收端轮询会消耗 SM 和内存流量，解包后还会增加一次本地写入。因此，Packet 更适合小消息、细粒度流水和边到达边消费，而普通 put/get 更适合追求大块数据的有效吞吐。

这里还需要特别区分 GPU 与 NPU 的高吞吐编程模式。CUDA 代码表面上可能只是每个线程读取一个 16B 值再写出，但硬件会以 Warp 为调度单位，把 32 个 Lane 的连续访问合并，并通过大量 Warp 形成高并发请求。若把同样的逐元素 GetValue、SetValue 写法直接翻译到昇腾 NPU，通常不会自动获得相同的请求组织方式。NPU 上更合理的做法是按 Tile 描述一批数据，利用 MTE 在 GM 与 UB 之间形成 Burst 搬运，或者在跨卡大块传输时使用 SDMA 等专用数据通路。也就是说，MemoryChannel 的算法语义可以迁移，但底层请求生成方式不能只做语法替换。

数据搬运完成之后，还必须解决通知和可见性问题。标准 signal 使用带 Release 语义的系统范围原子操作递增对端 Token，它要求此前的 put 写入在通知之前完成并对接收端可见；接收端 wait 使用 Acquire 语义轮询本地 inboundToken，只有观察到预期值后，后续读取才可以安全消费对应数据。relaxedSignal 和 relaxedWait 只适合控制执行进度，不会自动为其他普通内存访问建立完整顺序。如果协议需要“数据先写完、再通知、对端看到通知后再读”，就必须使用正确的 Release/Acquire 配对，或者显式补充所需 Fence。

跨设备同步之外，同一个 Kernel 内多个 Block 之间还需要 DeviceSyncer 建立阶段边界。__syncthreads 只能同步一个 Block，无法保证其他 Block 已经完成自己的数据分片。DeviceSyncer 会先在各 Block 内同步，再由每个 Block 的代表线程对全局计数器执行原子递增并等待所有参与 Block 到齐，最后再次放行本 Block 的其他线程。典型流程是由线程 0 与远端完成握手，经过一次 DeviceSyncer 后让全部线程共同执行 put 或 get，再经过第二次 DeviceSyncer，确认所有线程和 Block 都完成搬运，最后才由线程 0 发出完成 Signal。这样可以避免通知提前到达，而实际数据仍有一部分留在其他线程或 Block 的执行过程中。

总结来看，MemoryChannel 的核心优势是路径短：GPU 线程直接针对远端映射显存产生访问请求，普通 put/get 追求并行搬运和有效带宽，Packet 路径通过 Payload 加 Flag 追求小消息和细粒度流水，而 Semaphore 与 DeviceSyncer 分别负责跨设备和设备内的阶段同步。不过，这种直接远端访存依赖节点内可映射、可 P2P 访问的物理互联。当通信范围扩展到跨节点后，GPU 通常不能继续把整个网络传输当作普通远端 Global Store 来完成，这时就需要下一节的 PortChannel，把 GPU 的通信意图、CPU 的控制提交和 RDMA 网卡的数据搬运拆分开来。

## 12. PortChannel

前面我们介绍了 MemoryChannel。它面向节点内通信，由 GPU 线程直接对远端映射显存执行 load 和 store，核心目标是缩短数据路径并降低通信时延。不过，当通信范围从节点内扩展到跨节点之后，GPU 通常无法再像访问本地 NVLink 映射内存一样，直接完成整个网络传输流程。此时，MSCCL++ 通过 PortChannel 将 GPU 侧的通信意图、CPU 侧的传输控制以及网卡侧的数据搬运进行解耦，从而构建一条适用于 InfiniBand、RoCE 等网络传输场景的通信路径。

这一页首先给出了 PortChannel 的核心工作模式，可以概括为“GPU 出意图，CPU 出控制，RDMA 出搬运”。GPU 在 Kernel 内调用 put、putWithSignal 或 putWithSignalAndFlush 时，并不由 GPU 线程逐字节搬运 Payload，而是向共享 FIFO 中写入一条 128-bit 的 TriggerData。Trigger 中记录了本次操作的类型、源和目标偏移、传输长度以及是否需要 Signal、Flush 等控制信息。这样一来，GPU 只负责生成一条轻量级的通信命令，不需要退出 Kernel，也不需要由 Host 重新发起一次高层通信调用。

Trigger 写入 FIFO 后，运行在 CPU 侧的 ProxyService 会持续轮询该队列。为了减少跨 NUMA 访问和线程调度带来的额外开销，ProxyService 通常会绑定到靠近网卡的 CPU 核心，并采用 busy-poll 方式及时获取 GPU 提交的通信请求。ProxyService 解析 Trigger 后，通过对应的 Connection 调用 conn.write，将数据传输请求下发给 InfiniBand、RoCE 或其他传输后端。真正的 Payload 搬运由 RDMA NIC 通过 DMA 独立完成，数据从发送端注册内存直接写入接收端注册内存，搬运过程中不需要 CPU 执行数据拷贝，也不会占用 GPU 的计算线程。

从接口语义来看，put 只描述一次基础的远端写操作，ProxyService 解析后下发 RDMA Write。putWithSignal 在数据写之外增加了完成通知语义：ProxyService 先提交 Payload 的 RDMA 写请求，再通过 semaphore.signal 将 Token 写入对端的 inboundToken。接收端 GPU 在 wait 中使用 Acquire 语义轮询该 Token，只有在观察到预期值之后，才继续读取已经到达的远端数据。因此，Signal 和 Wait 解决的是发送端与接收端之间的跨节点通知和数据可见性问题，它建立了“数据先完成、通知后到达、接收方再读取”的同步关系。

这里还需要区分 Signal/Wait 与 Flush。Signal/Wait 面向通信两端，用于通知对端数据已经具备可见性；Flush 则主要用于协调本地 GPU 与 Host Proxy。GPU 提交 TriggerSync 后，会通过 fifo_.sync 等待 FIFO 中此前的请求被 ProxyService 消费。ProxyService 随后调用 conn.flush，推动已经提交到传输层的 RDMA 操作完成或达到后端定义的可见性要求。也就是说，FIFO 排空只能说明 Host 已经取走并处理了 Trigger，而 Connection Flush 进一步约束了底层传输队列的完成状态。putWithSignalAndFlush 将数据写、对端通知和本地完成确认组合在一起，适用于需要严格确认本轮通信已经推进完成的场景。

从页面中的结构图可以看到，PortChannel 形成了清晰的三级流水线。GPU 侧负责在 Kernel 中生成 Trigger，并将其写入 Device 与 Host 共享的 FIFO；CPU 侧的 ProxyService 负责解析命令、调用 conn.write、semaphore.signal 和 conn.flush；NIC 侧负责执行实际的 DMA 数据搬运。FIFO 的 head 位于 Device 侧，tail 由 Host 侧维护，这种生产者—消费者组织方式使 GPU 能够连续提交通信意图，同时由 ProxyService 按顺序转化为网络操作。

时序图进一步展示了 putWithSignalAndFlush 的完整执行过程。发送端 GPU 首先向 FIFO 写入同时包含 Data、Flag 和 Sync 语义的 Trigger，随后进入 fifo_.sync 等待。ProxyService 轮询到该 Trigger 后，先通过 conn.write 下发 Payload 的 RDMA 写，NIC 将数据直接搬运到对端显存；然后 ProxyService 调用 semaphore.signal，通过 RDMA 写更新对端 inboundToken；最后执行 conn.flush，并在相关请求处理完成后推进 FIFO 状态，使发送端 GPU 的同步等待得以返回。与此同时，对端 GPU 在 wait 中以 Acquire 语义观察 Token，在 Token 到达后即可安全读取对应数据。

因此，与上一节 MemoryChannel 的 GPU 直接远端访存模式相比，PortChannel 并不是让 CPU 重新承担数据拷贝，而是让 CPU Proxy 承担网络控制面的提交工作，将实际数据面交给 RDMA NIC。它在保留 GPU-driven 编程入口的同时，适配了跨节点网络所需要的队列管理、传输提交和完成语义，是 MSCCL++ 面向大消息和多机通信的重要通道。接下来，我们再看第三种 Channel——SwitchChannel。它不再依赖 GPU、CPU Proxy 和 NIC 之间的软件协同，而是进一步利用 NVLS 和 multimem 指令，将部分集合通信归约操作直接下沉到交换结构硬件中完成。

## 13. SwitchChannel

上一节介绍的 PortChannel，通过 GPU、CPU Proxy 和 RDMA NIC 的三级协作解决了跨节点数据传输问题。SwitchChannel 则采用了另一种优化思路：当系统具备 NVLS，也就是 NVLink SHARP 相关硬件能力时，不再由软件组织多轮点对点传输来完成归约，而是由 GPU 线程直接发出 multimem 指令，把跨 GPU 的数据聚合和分发操作下沉到 NVLS 交换结构中执行。因此，这一页关注的重点不是如何提交一次网络搬运，而是如何利用互联硬件本身提供的集合操作能力，缩短归约路径并减少软件调度开销。

SwitchChannel 的核心价值，可以概括为“一条 GPU 指令触发一次硬件级集合操作”。以 reduce 接口为例，GPU 线程执行 multimem.ld_reduce 指令后，NVLS 会访问参与 GPU 对应位置的数据，在交换结构内部完成指定的归约运算，并将归约结果直接返回给发起访问的 GPU。与传统 Ring AllReduce 需要经过多个通信轮次、逐跳传递数据相比，这种方式把原本由多个 send、recv 和 reduce 步骤组成的过程，压缩为一次面向 multimem 地址的硬件归约操作。

从接口层面来看，reduce、broadcast 和 multimemStoreReduce 分别对应三类典型能力。reduce 通过 multimem.ld_reduce 在读取数据的同时完成跨 GPU 归约；broadcast 通过 multimem.st 将一个值写入所有参与 GPU 对应的内存位置；multimemStoreReduce 则通过 multimem.red 在交换结构中完成归约，并将结果写回相关 GPU。也就是说，SwitchChannel 不只是提供数据搬运接口，而是直接暴露了带有集合语义的硬件操作，使归约和广播不再需要由软件拆解为多次普通内存访问。

页面中的结构图展示了这一过程。多个 GPU 分别针对同一个集合内存位置发起 multimem 操作，请求进入 NVLS Switch 后，由交换结构同步读取各个 GPU 的输入并执行归约。硬件得到最终结果后，再将结果返回到相应 GPU。这里最关键的变化，是数据不需要先集中到某个 GPU 上由 SM 执行归约，也不需要沿 Ring 拓扑经过多轮转发；交换结构本身就是归约执行单元，因此能够显著减少数据往返次数和中间状态。

时序图进一步强调了 SwitchChannel 的并行性。GPU0、GPU1 以及其他参与 GPU 可以面向同一个 multimem 地址发起归约指令，NVLS 在硬件内部聚合这些输入，并将结果返回给请求方。从 GPU Kernel 的视角来看，开发者使用的是一条 load-reduce 或 store-reduce 指令；但从硬件数据路径来看，这条指令实际表达的是一次跨多个 GPU 的集合操作。这也是 SwitchChannel 与 MemoryChannel、PortChannel 最本质的区别：前两者主要提供远端数据访问和传输能力，而 SwitchChannel 进一步把归约语义下沉到了互联硬件。

在数据类型和访存粒度方面，SwitchChannel 支持 f32、f16、bf16、FP8、i32、u32 和 f64 等多种类型，并能够使用 x1、x2、x4、x8 或 x16 等向量化形式进行操作。向量化能够让一条指令处理更多连续数据，但实际可用的数据类型、向量宽度和归约操作仍然受到具体 GPU 架构与 NVLS 指令能力的约束。因此，在设计 Kernel 时，需要同时考虑数据布局、地址对齐以及目标硬件能够支持的 multimem 指令组合。

需要注意的是，SwitchChannel 对内存有更严格的要求。mcPtr 必须指向 multimem-addressable 内存，使多个 GPU 能够通过统一的集合地址访问各自对应的数据；devicePtr 则通常用于保存本地结果或承接归约输出。这意味着普通的独立显存指针不能直接替代 multimem 内存，运行时需要完成特定的内存分配、映射和注册，并保证参与 GPU 之间满足相应的对称地址或特殊内存布局要求。

另外，页面中列出的 multimem 指令采用 relaxed.sys 语义。Relaxed 只保证指令本身的原子性或操作语义，并不会自动为前后的普通内存访问建立完整的顺序关系。因此，当归约结果依赖此前写入的数据，或者后续计算必须等待归约结果稳定可见时，Kernel 仍然需要根据生产者—消费者关系插入合适的 fence、barrier 或其他同步操作。硬件完成归约，并不等同于程序天然获得了完整的 Release/Acquire 顺序保证。

最后，SwitchChannel 是一条与硬件能力强相关的高性能路径。它依赖 NVLS、NVLink SHARP 以及可被 multimem 访问的内存资源，并不是所有 GPU 型号和系统拓扑都能够使用。当运行环境不具备这些条件时，MSCCL++ 需要回退到 MemoryChannel、PortChannel，或者由 NCCL 等通用通信路径完成相同的集合操作。因此，SwitchChannel 的意义并不是替代所有其他 Channel，而是在硬件条件满足时，为归约和广播提供一条更短、更直接的软件到硬件映射路径。

通过这三种 Channel 可以看到，MSCCL++ 会根据通信范围和硬件能力选择不同的数据路径：MemoryChannel 面向节点内直接访存，PortChannel 面向跨节点 RDMA 传输，SwitchChannel 面向 NVLS 硬件集合操作。理解这些通道之后，下一步还需要继续关注它们共同依赖的内存基础，也就是 RegisteredMemory、远端访问元数据以及对称 Offset 等内存布局约束。

## 15. RegisteredMemory 与对称 Offset

前面三种 Channel 虽然采用了不同的数据路径，但它们都有一个共同前提：通信端不能只拿到一段普通内存的本地指针，还必须知道这段内存是否能够被相应的 Transport 访问，以及远端应该通过什么方式定位和操作它。因此，这一页从 Channel 的执行机制进一步下沉到内存基础，重点介绍 RegisteredMemory 和对称 Offset。可以先给出一个核心结论：MSCCL++ 中的 RegisteredMemory 不是单纯的地址封装，而是一份同时包含本地内存范围和远端访问元数据的通信资源描述。

对于本地程序来说，一个裸指针通常只能说明虚拟地址从哪里开始，却不能直接支持跨进程、跨设备或者跨节点访问。不同 Transport 需要不同类型的附加信息。例如，CudaIpc 路径需要能够在另一个进程中重新打开显存映射的 IPC Handle；InfiniBand 或 RoCE 路径需要完成 Memory Region 注册，并生成网卡执行 RDMA 所需的 local key、remote key 以及地址范围；NVLS 路径则要求内存具备 multimem-addressable 属性。RegisteredMemory 的作用，就是把这些与 Transport 相关的信息和原始 Buffer 关联起来，使 Connection 和 Channel 能够基于统一对象构造远端访问能力。

从通信建立流程来看，发送端和接收端会先分别注册本地 Buffer，再通过控制面交换内存描述符。交换的不是实际 Payload，而是远端访问这段内存所需要的 Handle、Key、基地址、长度以及其他 Transport Metadata。连接建立后，Channel 会把这些信息进一步整理为 GPU Kernel 或 ProxyService 可以使用的 Device Handle。这样，在真正执行 put、get、RDMA Write 或 multimem 操作时，数据面不需要再次经过 Host 中转，也不需要临时查询远端内存信息，而是可以直接根据预先建立的访问描述完成传输。

这一页还强调了对称 Offset 的概念。所谓对称，并不一定要求不同 Rank 上的 Buffer 具有完全相同的虚拟地址，而是要求同一逻辑数据块在各 Rank 的注册内存区域中具有一致的相对偏移。例如，chunk1 在 Rank0、Rank1 和 Rank2 的 Buffer 中都位于相对于各自基地址的同一个 Offset。这样，通信算法只需要描述“访问第几个 Chunk”或者“从哪个 Offset 开始”，运行时就可以使用远端基地址加统一 Offset 的方式定位对应数据，而不必为每个 Rank 维护一套完全不同的地址计算逻辑。

图中的多个 Rank 都按照相同顺序组织 chunk0、chunk1、chunk2 和 chunk3，其中同名 Chunk 处于相同的相对位置。假设算法需要访问 chunk1，那么对于任意目标 Rank，都可以复用同一个 Offset，只需要替换该 Rank 对应的 RegisteredMemory 基址和访问元数据。这种布局能够显著简化集合通信算法中的地址生成，也便于 DSL、Executor 和 Primitive 层把逻辑 Chunk 映射到具体的远端内存位置。对于 AllGather、ReduceScatter 等规则分片的集合通信，对称布局尤其重要。

对称 Offset 还能够降低设备端控制信息的体积。如果每次通信都必须传递完整的远端绝对地址，Channel Handle 和 Trigger 中就需要携带更多针对单个 Rank 的地址数据；采用“RegisteredMemory 基址加 Offset”的方式之后，算法只需要传递较小的偏移和长度信息，远端地址可以由既有元数据快速计算。这也解释了为什么 PortChannel 的 Trigger 可以重点描述 Offset 和 Size，而不需要在每次操作中重新完成完整的内存注册与地址交换。

不过，zero-copy 并不意味着没有成本，而是把一部分运行时拷贝成本转化成了更严格的初始化和布局约束。应用需要提前确定 Buffer 的生命周期和大小，完成注册与元数据交换，并在通信期间保证内存不会被释放、重新分配或越界访问。对于需要对称布局的算法，各 Rank 还必须对 Chunk 顺序、Offset、对齐方式和容量形成一致约定。只要其中一个 Rank 的布局不同，相同 Offset 就可能指向错误数据，甚至造成非法远端访问。

因此，RegisteredMemory 同时承担了性能和安全边界两方面的职责。从性能角度看，它让 GPU、NIC 或 NVLS 能够直接访问预注册的用户 Buffer，减少中间拷贝和重复控制面操作；从正确性角度看，它限定了远端可以访问的内存范围，并提供 Transport 验证访问权限所需的 Handle 或 Key。实际使用时，还需要把 RegisteredMemory 的生命周期与 Connection、Channel 以及正在执行的 Kernel 或 RDMA 请求协调起来，确保所有异步通信完成之后才能注销或释放对应内存。

总结来看，MSCCL++ 的 zero-copy 能力并不是简单地绕过一次 memcpy，而是建立在“内存预注册、元数据预交换、地址按 Offset 解析、布局在各 Rank 间保持一致”这一整套机制之上。RegisteredMemory 解决远端能否访问的问题，对称 Offset 解决算法如何统一定位数据的问题，两者共同构成 MemoryChannel、PortChannel 和 SwitchChannel 的内存基础。在明确了这些底层资源之后，下一节就可以继续向上看 DSL 如何通过 Rank、Buffer、Chunk 和 Channel 等对象，把具体的内存位置和通信动作组织成可编译的集合通信算法。

## 21. XCCL++ DSL 入门：从算法描述到 NPU 执行

**21.** 前面几节已经把视角深入到 Channel、RegisteredMemory 和远端数据路径，解释了通信资源怎样建立、数据最终由谁搬运。接下来这一章把视角向上移动到 XCCL++ DSL：开发者不再从某一条底层搬运指令开始组织算法，而是先描述参与通信的 Rank、数据分片、Channel 关系和同步顺序，再由编译器与运行时把这些描述转换成 NPU 可以执行的计划。理解这一层的关键，是始终把“算法描述”“运行时调度”和“设备端搬运”区分开，同时又能看清它们之间的映射关系。

**21.1** DSL 是领域特定语言，XCCL++ DSL 针对的领域就是集合通信算法。它使用 Python 提供较高层的描述方式，让开发者表达哪些 Rank 参与、数据从哪个 Buffer 的哪个 Offset 移动到哪里、采用哪类 Channel，以及何时执行 signal、wait、put、get、reduce 或 broadcast。这里的 Python 代码更像是在构建一张通信执行蓝图，而不是逐句解释执行真实数据搬运；蓝图会先形成 CollectiveProgram，经过编译生成 JSON 执行计划，再由 C++ Runtime 和 Executor 调度 NPU Kernel 与底层 Channel 完成实际工作。

**21.2** 这一小节需要把 DSL 与前面讲过的 MemoryChannelDeviceHandle 联系起来。前面的 put、get、signal、wait 和 Packet 机制，回答的是设备端“具体怎样搬、怎样通知”；DSL 回答的则是算法层“为什么在这里搬、谁与谁通信、前后依赖怎样排列”。因此可以把系统理解成上下两层：上层 CollectiveProgram 决定算法的拓扑和步骤，下层 Channel、DeviceHandle、Kernel 以及 MTE、SDMA 或远端访存路径负责落实这些步骤。DSL 能减少重复样板代码，但不会替用户消除地址、对齐、容量、同步和物理拓扑方面的约束。

**21.3** 有了分层概念之后，再看 PPT 中的超节点拓扑就会更清楚。多个 NPU 先通过节点内或机框内的 UB 交换层连接，再进一步汇聚到更高层交换设备和外部网络，通信距离和硬件能力并不相同。XCCL++ 因此抽象出 MemoryChannel、PortChannel 和 SwitchChannel 三类通路：MemoryChannel 适合对远端映射内存进行直接读写，PortChannel 面向基于端口和 RDMA 的异步跨节点传输，SwitchChannel 则利用交换设备承担规约或多播。三者不是简单的 API 名称差异，而是在表达完全不同的数据路径和硬件分工。

**21.4** 整个 DSL 使用流程可以归纳为算法定义、初始化编译和算法调用三个阶段。开发者先用 AlgoSpec 声明适用条件，再在 CollectiveProgram 中创建 Channel、同步关系和数据操作；随后调用 compile，把 Python 构建出的蓝图转成 ExecutionPlan 和 Algorithm；运行时侧由 CommGroup 建立 Rank 拓扑，由 Executor 负责调度；真正调用集合通信时，再通过 Algorithm.execute 绑定本次 Buffer、大小、数据类型和 Stream。最后还可以用 CustomizedComm 把这些对象封装成 all_reduce、all_gather 一类更接近业务侧习惯的公共接口。

**21.5** 第一步是 AlgoSpec，它描述的是算法“在什么条件下可以运行”，而不是具体的数据搬运步骤。这里通常会声明算法名称、集合通信类型、全局 Rank 数、每节点 Rank 数、是否原地执行、协议类型、实例数、线程配置、Buffer 对齐以及消息大小范围。AlgoSpec 的价值在于把算法选择条件显式化：同一个 AllGather，在八张 NPU 和十六张 NPU 上、在小消息和大消息上、在 Simple 和 LL 协议下，可能需要不同的执行计划。可以把 AlgoSpec 看成算法的适用范围与资源配置合同。

**21.6** 第二步进入 CollectiveProgram，开始真正描述“数据怎样流动”。算法函数在 CollectiveProgram 上下文中构建一张有顺序、有依赖的通信图，通常依次完成 Channel 创建、通信就绪同步、数据操作和阶段完成同步。这里写下的操作不会立刻在 Python 进程里搬运数据，而是被记录为后续编译所需的节点和依赖。也就是说，CollectiveProgram 是连接算法思想与执行计划的核心中间表示，既包含 Rank 间的数据关系，也包含运行时必须遵守的阶段边界。

**21.6.1** 创建 Channel 时，开发者需要明确通信双方、参与组以及期望使用的硬件路径。对于点对点远端内存访问，可以为对应 Rank 对建立 MemoryChannel；跨节点异步传输可以使用 PortChannel；需要交换侧规约或多播时，则组织 SwitchChannel。这个步骤的本质，是把抽象的数据边绑定到可实现它的传输资源。Channel 选错并不只是性能下降，还可能意味着目标拓扑根本不支持对应操作，因此它是算法正确映射到硬件的第一道关口。

**21.6.2** Channel 建立之后通常要先处理“同步就绪”。signal 和 wait 在这里用于确认接收端 Buffer、远端映射、通信轮次或相关资源已经准备完成，避免发送方过早写入尚未可用的目标区域。讲解这一部分时要强调，同步不是附加装饰，而是数据协议的一部分：只有明确谁发出通知、谁等待哪个 Token，以及通知之前哪些写入必须完成，后续的数据操作才有可靠的起点。

**21.6.3** 完成就绪同步后，CollectiveProgram 才记录 put、get、copy、reduce、broadcast 等数据操作。put 表达本地到远端的数据推进，get 表达从远端拉取，copy 表达指定范围的数据复制，而 reduce 和 broadcast 进一步带有集合语义。DSL 中的一次调用描述的是一个逻辑执行阶段，编译和运行时可能把它展开为多个核、多个线程、多个分片甚至专用搬运引擎的协同执行，因此不能把一行 DSL 代码等同于一条硬件指令。

**21.6.4** 数据操作之后还需要“同步完成”，用于建立清晰的阶段边界。算法必须确认本轮分片是否全部到达、后续步骤是否依赖完整结果，以及相关 Buffer 是否已经可以复用。这个问题与前面介绍的 DeviceSyncer、Semaphore 和 Release/Acquire 语义是一脉相承的，只是 DSL 和编译器能够帮助组织一部分依赖。用户仍然需要保证图中表达了真实的生产者—消费者关系，尤其要注意跨核、跨进程和跨设备场景下的完成与可见性并不是同一个概念。

**21.7** AlgoSpec 与算法函数是相互配合、职责不同的两部分。AlgoSpec 规定舞台规模和运行规则，例如 Rank 数、消息范围、协议和实例配置；算法函数与 CollectiveProgram 则规定演员如何走位，也就是 Rank 之间怎样连接、Offset 如何计算、何时 signal 和 wait、何时执行具体数据操作。两者都会影响最终编译结果和缓存键，但不要混淆：调整消息范围通常属于 Spec 变化，调整 Ring 的连接方向或分片搬运顺序则属于算法图变化。

**21.8** 第三步是 CommGroup，它负责让分布在不同进程中的 Rank 彼此认识并建立统一通信上下文。初始化时，各 Rank 通过 TCP Bootstrap 向默认 Master 汇报进程、节点和设备信息，Master 汇总完整拓扑后再广播给所有参与者，随后各进程创建一致的 C++ Communicator 视图，并完成 Rank 与 NPU 的绑定。可以把 CommGroup 理解为通信系统的签到、组网和通讯录分发阶段；如果这里的 Rank、网卡、地址或设备绑定错误，后面的算法计划即使完全正确也无法正常执行。

**21.9** 第四步调用 xcclpp.compile，把算法函数与 AlgoSpec 转换成可执行计划。编译器首先运行 Python 构图逻辑得到 CollectiveProgram，再将其序列化为 JSON，并进行依赖解析、同步处理、实例复制、资源规划和必要优化；随后把结果写入缓存，并由 C++ 侧创建对应 Rank 的 ExecutionPlan，最终返回 Algorithm 对象。缓存键通常同时受算法源码和 Spec 元数据影响，所以 Rank 数、协议、消息范围、实例数、对齐或线程配置发生变化时，都应当被视为不同的编译条件，而不能盲目复用旧计划。

**21.10** 第五步是 Executor，它是把静态执行计划真正送到 NPU 上运行的调度者。CommGroup 提供通信上下文，Algorithm 保存编译后的图纸，而 Executor 负责根据图纸准备 Channel、启动 Kernel、推进执行阶段并协调相关运行时资源。用一个直观类比来说，Algorithm 是施工图，Communicator 是工地的通信网络和人员名册，Executor 是现场总调度，NPU Kernel、MTE、SDMA 和各种 Channel 才是实际完成搬运与计算的设备。

**21.11** 第六步通过 Algorithm.execute 绑定本次调用的真实数据。编译阶段得到的 Algorithm 更像可复用模板，execute 才传入输入输出 Buffer 的 NPU 地址、字节数、数据类型、规约操作、Stream、执行块配置以及其他算法参数。这里尤其要检查“元素数量”和“字节数量”是否混用，Buffer 是否满足对齐和容量要求，Stream 是否与外部计算建立正确顺序。只要仍满足 AlgoSpec 的约束，同一个 Algorithm 可以多次执行，并在每次调用时使用不同的 Buffer 地址和 Stream。

**21.12** 第七步是 CustomizedComm，它把 CommGroup、Executor 和一组 Algorithm 再封装成业务侧熟悉的公共接口。上层调用者最终只需要面对 all_gather、all_reduce 等方法，而不必在每次通信时重复管理 communicator、execution plan 和 execute 参数。这个封装并没有创造新的底层能力，它的价值在于统一算法选择、参数检查、Stream 转换和对象生命周期，使自定义 DSL 算法能够以接近 torch.distributed 的方式接入现有训练或推理框架。

**21.13** 把前面的步骤合在一起，就得到一个从定义到执行的最小骨架：先构造 AlgoSpec，再编写返回 CollectiveProgram 的算法函数；函数内部创建 Channel、建立开始同步、记录数据操作并设置完成同步；运行时初始化 CommGroup，调用 compile 得到 Algorithm，再创建 Executor；业务执行时，通过 Algorithm.execute 传入真实 Tensor 地址和 Stream。这个骨架的重点不在于记住某段示例代码，而在于明确对象的生命周期：Spec 和算法函数用于构图，Algorithm 可被复用，CommGroup 与 Executor 服务于运行时，而 Buffer 属于每一次具体调用。

**21.14** 接下来要回答最重要的问题：DSL 操作最终怎样落到底层。以 MemoryChannel.put 为例，Python 侧先记录源 Rank、目标 Rank、Buffer、Offset 和长度，compile 将其转成 ExecutionPlan 中的 Channel 指令，Executor 再启动设备端阶段并取得相应 DeviceHandle，最后由 NPU 核、MTE、SDMA 或适配后的远端内存访问路径完成实际搬运。由此可见，DSL 负责表达意图和依赖，执行计划负责资源化和排序，设备端实现负责产生真实内存请求；分析性能时必须沿着这条链路继续向下追踪，不能停留在 Python API 名称上。

**21.15** Simple 与 LL 协议体现了不同的优化目标。Simple 更强调传输纯 Payload，并通过独立同步机制确认阶段完成，通常更适合大块数据和较高有效带宽；LL 则把 Payload 与 Flag 或轮次信息结合，让接收端能够细粒度轮询并边到达边消费，更偏向小消息和低延迟。LL 的代价是额外元数据、轮询流量、较低的 Payload 利用率以及更复杂的 Flag 回绕管理。因此协议选择必须同时观察消息大小、同步频率、流水需求、链路带宽和接收端执行资源。

**21.16** auto_sync 只能理解为“编译器根据已知依赖自动组织部分同步”，不能理解为“不再需要考虑同步”。编译器无法替用户推断任意裸指针访问、外部 Kernel、跨程序阶段或多个生产者对同一计数器的非原子更新。使用 auto_sync 时仍要明确谁生产、谁消费，Signal 前是否保证相关写入完成，Wait 后是否建立所需可见性，以及依赖是否完整地位于同一个 CollectiveProgram 中。自动化能够减少机械工作，但不能替代对内存语义和并发协议的判断。

**21.17** DSL 出现问题时，排查顺序应从上到下逐层收敛。先确认 AlgoSpec 是否匹配当前 world_size、消息大小、原地模式和协议，再检查 CollectiveProgram 中的 Rank、Buffer、Offset、长度及 signal、wait 配对；随后确认 Channel 是否符合物理拓扑，compile 生成的 JSON 是否符合预期，CommGroup 组网与设备绑定是否正确，Executor 和 communicator 是否对应，最后再进入 execute 参数、设备 Kernel、MTE、SDMA 或远端访存路径。按照层次排查，可以避免一开始就陷入底层日志，却忽略了上层计划本身已经写错。

**21.18** 最终需要建立的是一套完整的分层心智模型：AlgoSpec 定义任务规模和规则，CollectiveProgram 描述线路与操作顺序，三类 Channel 选择运输通路，CommGroup 建立参与者拓扑，compile 把蓝图转换成工单，Algorithm 保存可复用模板，Executor 负责现场调度，execute 为本次任务填入真实 Buffer，而 DeviceHandle 和 Kernel 在设备端落实地址访问、同步和数据搬运。学习 XCCL++ DSL 时不要只背 API，要持续追问每个操作最终生成了什么阶段、数据物理上从哪里到哪里、使用哪条硬件路径，以及完成和可见性究竟由谁保证。只有这样，DSL 才能与 MemoryChannel、Packet、MTE、SDMA 和集合通信算法真正连成一个体系。

## 20. NCCL API 兼容层

前面已经把 DSL 从算法描述、编译计划一直讲到 Executor 和设备端 Primitive，但在真实训练和推理系统中，应用通常不会直接管理这些内部对象。现有框架已经大量使用 ncclAllReduce、ncclAllGather 和 ncclReduceScatter 等标准接口，如果为了引入 MSCCL++ 而要求业务代码整体改写，接入成本和风险都会非常高。因此，NCCL API 兼容层解决的核心问题不是重新定义一套集合通信接口，而是在尽量保持上层调用方式不变的前提下，把实际执行路径切换到更适合当前场景的 MSCCL++ 算法。

这条路径的入口是 Interposition，也就是对 NCCL API 调用进行拦截。应用仍然发起熟悉的 ncclAllReduce 或 ncclAllGather，兼容层可以通过 LD_PRELOAD、审计 Shim 或类似的动态链接机制接管调用，然后把本次 collective 的类型、Buffer、消息大小、数据类型、规约操作、Communicator 和 Stream 等信息交给 Algorithm Selector。对上层应用来说，函数签名和调用习惯没有改变；变化发生在 API 之下，原本直接进入 NCCL 内部实现的请求，现在多了一次可控的路径选择。

Selector 首先要判断本次请求是否具备使用优化算法的条件。它会识别当前是 AllReduce、AllGather 还是 ReduceScatter，检查消息规模是否落在某个算法的适用范围内，再结合节点内或跨节点拓扑、NVLink、InfiniBand、NVLS 等硬件能力，判断哪些数据路径实际可用。除此之外，还要确认相应的 DSL JSON Plan、Native Kernel 或已初始化资源是否存在，以及 Buffer、协议、Rank 数等运行参数能否匹配算法要求。也就是说，Selector 不是只按照算子名称做静态分支，而是在调用现场完成一次算法约束匹配。

当条件满足时，Selector 可以选择三类执行路径。第一类是 DSL Plan，由 Executor 加载预先编译的 JSON 执行计划，并按照其中的 Channel、操作和同步依赖启动 Kernel；第二类是 Native Kernel，也就是直接进入针对特定场景手写的 C++ 或 CUDA 高性能实现；第三类是 NCCL fallback，当当前请求没有合适计划、硬件能力不足、参数不匹配，或者系统处于保守运行模式时，继续调用原始 NCCL 完成通信。三条路径对外保持相同的 collective 语义，但内部的数据流、调度方式和优化空间并不相同。

这里需要强调，兼容层的目标不是无条件替换 NCCL，而是提供一个可以渐进启用、随时回退的优化入口。通信属于分布式系统中的关键正确性路径，任何 Rank 数、消息范围、Buffer 布局、Stream 顺序或硬件能力判断错误，都可能造成错误结果或永久等待。因此，调试阶段通常需要支持按算子、消息范围或算法标识强制 fallback，以便将 MSCCL++ 路径与 NCCL baseline 对比；生产阶段也需要在没有可靠匹配项时优先保证正确性，而不是为了命中优化路径而勉强执行不适用的算法。

从系统集成角度看，这一层把应用生态和底层通信创新解耦开来。PyTorch、LLM Serving 框架或其他上层系统继续使用稳定的 NCCL 风格接口，MSCCL++ 则可以在兼容层内部逐步增加新的 DSL Plan、Packet 小消息路径、NVLS 算法或 Native Kernel。应用不需要感知每一种 Channel 和执行计划的细节，但仍然能够获得针对消息形态和机器拓扑定制的通信路径。这也是兼容层的重要价值：它让算法优化能够在较低迁移成本下进入现有生产系统。

不过，完成 API 拦截只解决了“请求怎样进入 MSCCL++”的问题，还没有回答“同一个 collective 应该选择哪一个具体算法”。实际系统中，一个 AllReduce 往往同时存在面向小消息、节点内全互联、分阶段 RSAG、流水化以及 NVLS 硬件加速的多种实现。Selector 要想作出合理决策，就必须面对下一节介绍的 Algorithm Collection，也就是先把同一 collective 的多个候选算法组织成一个可管理、可匹配的算法族。

## 21. Algorithm Collection 与 Selector

这一节进一步说明，为什么一个 collective 不能只绑定一个固定实现。AllReduce 的数学语义始终是对各 Rank 数据进行规约，并让所有参与者获得结果，但完成这项工作的通信路径可以有很大差异。小消息更关注固定延迟，大消息更关注链路带宽；单节点可以利用 GPU 间直接访问，多节点需要考虑 RDMA；具备 NVLS 的系统可以把部分规约下沉到交换硬件，而普通系统仍需依赖点对点传输。因此，MSCCL++ 会为同一种 collective 维护一组候选实现，而不是假设存在一个在所有条件下都最优的算法。

以 AllReduce Algorithm Collection 为例，packet 算法通常面向小消息，通过 Data 与 Flag 结合减少独立同步开销；fullmesh 更适合节点内 GPU 之间连接充分的场景，让 Rank 直接交换和处理数据；rsag 将 AllReduce 拆分为 ReduceScatter 和 AllGather 两个阶段，利用规则分片降低每一阶段的数据量；rsag_pipeline 在此基础上继续把 Chunk 划分为多个流水阶段，使通信、规约和后续转发能够重叠推进；nvls_zero_copy 利用 NVLS 和可直接访问的内存缩短硬件归约路径；nvls_packet 则进一步把 NVLS 能力与小消息 Packet 协议结合。它们对外完成相同语义，但优化目标和硬件前提各不相同。

Algorithm Collection 的作用，就是把这些算法按统一方式进行登记和组织。每一个候选项不仅需要关联具体 Kernel 或 ExecutionPlan，还要携带自己的适用条件，例如 collective 类型、Rank 数、节点数量、消息大小上下限、协议、数据类型、是否原地执行、拓扑要求以及是否依赖 NVLS。这样，Selector 面对一次真实调用时，不需要在大量实现代码中临时猜测，而是可以先从 Collection 中筛选满足约束的候选集合，再按照优先级、经验规则或性能策略选出最终算法。

消息大小通常是最直接的选择维度。对于很小的数据，Kernel 启动、同步和协议往返等固定开销占比很高，因此 Packet、融合操作或专门的小消息 Kernel 往往更有优势；随着消息变大，有效 Payload 比例和链路利用率变得更重要，FullMesh、RSAG 或流水化算法可能逐渐占优。这里不存在一个固定的全局分界点，因为阈值还会受到 GPU 数量、链路带宽、拓扑层次、线程配置和实现成熟度影响，所以算法适用范围需要与具体部署环境共同校准。

拓扑和硬件能力决定了候选算法是否真正可执行。单节点全互联环境可以充分利用 MemoryChannel 和 P2P 映射，但跨节点通信通常必须引入 PortChannel 或其他网络路径；NVLS 算法只有在硬件、驱动和内存分配方式均满足要求时才应进入候选集合。换句话说，Selector 不能只看到算法名称，还必须理解算法背后的资源假设。一个在 NVLink 全互联机器上表现优秀的 FullMesh 算法，放到存在 PCIe 绕行或跨节点链路的系统上，可能不仅性能下降，还会因访问路径不可用而无法执行。

Collection 还承担初始化资源和缓存复用方面的职责。构造算法时往往需要建立 Communicator、Connection、RegisteredMemory、Channel、Executor 或 ExecutionPlan 等运行时对象，这些准备工作不适合在每次 collective 调用时重复完成。系统通常会根据 Communicator、拓扑、Rank 数、协议和算法参数形成 Context Key，把已经构建好的算法实例或相关资源缓存起来。后续请求命中相同条件时，就可以复用已有上下文，避免把控制面初始化成本重新带入高频通信路径。

从完整执行链来看，应用首先通过 NCCL 兼容接口提交 collective，Interposition 将请求转给 Selector；Selector 根据本次参数从 Algorithm Collection 中筛选候选项，检查消息范围、拓扑、硬件能力和资源状态，然后选中 DSL Plan、Native Kernel 或其他专用实现；如果没有任何候选项能够安全匹配，则回到 NCCL fallback。因而 Collection 负责回答“系统有哪些可选算法”，Selector 负责回答“这一次应该用哪一个”，而具体 Kernel 和 Primitive 则负责“选中之后怎样真正执行”。

总结来看，Algorithm Collection 与 Selector 共同构成了 MSCCL++ 的运行时决策层。Collection 让不同消息规模、拓扑和硬件条件下的算法能够同时存在，并以统一约束进行管理；Selector 则在保持 collective 语义不变的前提下，为每一次调用选择更合适的执行路径。理解这一层之后，就能看到 MSCCL++ 的优化并不是简单用一个新 Kernel 替换 NCCL，而是从兼容入口、候选算法管理、运行时匹配，到 Channel 和硬件执行形成一条完整的自适应通信链路。

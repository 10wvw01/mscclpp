## 12. PortChannel

前面我们介绍了 MemoryChannel。它面向节点内通信，由 GPU 线程直接对远端映射显存执行 load 和 store，核心目标是缩短数据路径并降低通信时延。不过，当通信范围从节点内扩展到跨节点之后，GPU 通常无法再像访问本地 NVLink 映射内存一样，直接完成整个网络传输流程。此时，MSCCL++ 通过 PortChannel 将 GPU 侧的通信意图、CPU 侧的传输控制以及网卡侧的数据搬运进行解耦，从而构建一条适用于 InfiniBand、RoCE 等网络传输场景的通信路径。

这一页首先给出了 PortChannel 的核心工作模式，可以概括为“GPU 出意图，CPU 出控制，RDMA 出搬运”。GPU 在 Kernel 内调用 put、putWithSignal 或 putWithSignalAndFlush 时，并不由 GPU 线程逐字节搬运 Payload，而是向共享 FIFO 中写入一条 128-bit 的 TriggerData。Trigger 中记录了本次操作的类型、源和目标偏移、传输长度以及是否需要 Signal、Flush 等控制信息。这样一来，GPU 只负责生成一条轻量级的通信命令，不需要退出 Kernel，也不需要由 Host 重新发起一次高层通信调用。

Trigger 写入 FIFO 后，运行在 CPU 侧的 ProxyService 会持续轮询该队列。为了减少跨 NUMA 访问和线程调度带来的额外开销，ProxyService 通常会绑定到靠近网卡的 CPU 核心，并采用 busy-poll 方式及时获取 GPU 提交的通信请求。ProxyService 解析 Trigger 后，通过对应的 Connection 调用 conn.write，将数据传输请求下发给 InfiniBand、RoCE 或其他传输后端。真正的 Payload 搬运由 RDMA NIC 通过 DMA 独立完成，数据从发送端注册内存直接写入接收端注册内存，搬运过程中不需要 CPU 执行数据拷贝，也不会占用 GPU 的计算线程。

从接口语义来看，put 只描述一次基础的远端写操作，ProxyService 解析后下发 RDMA Write。putWithSignal 在数据写之外增加了完成通知语义：ProxyService 先提交 Payload 的 RDMA 写请求，再通过 semaphore.signal 将 Token 写入对端的 inboundToken。接收端 GPU 在 wait 中使用 Acquire 语义轮询该 Token，只有在观察到预期值之后，才继续读取已经到达的远端数据。因此，Signal 和 Wait 解决的是发送端与接收端之间的跨节点通知和数据可见性问题，它建立了“数据先完成、通知后到达、接收方再读取”的同步关系。

这里还需要区分 Signal/Wait 与 Flush。Signal/Wait 面向通信两端，用于通知对端数据已经具备可见性；Flush 则主要用于协调本地 GPU 与 Host Proxy。GPU 提交 TriggerSync 后，会通过 fifo_.sync 等待 FIFO 中此前的请求被 ProxyService 消费。ProxyService 随后调用 conn.flush，推动已经提交到传输层的 RDMA 操作完成或达到后端定义的可见性要求。也就是说，FIFO 排空只能说明 Host 已经取走并处理了 Trigger，而 Connection Flush 进一步约束了底层传输队列的完成状态。putWithSignalAndFlush 将数据写、对端通知和本地完成确认组合在一起，适用于需要严格确认本轮通信已经推进完成的场景。

从页面中的结构图可以看到，PortChannel 形成了清晰的三级流水线。GPU 侧负责在 Kernel 中生成 Trigger，并将其写入 Device 与 Host 共享的 FIFO；CPU 侧的 ProxyService 负责解析命令、调用 conn.write、semaphore.signal 和 conn.flush；NIC 侧负责执行实际的 DMA 数据搬运。FIFO 的 head 位于 Device 侧，tail 由 Host 侧维护，这种生产者—消费者组织方式使 GPU 能够连续提交通信意图，同时由 ProxyService 按顺序转化为网络操作。

时序图进一步展示了 putWithSignalAndFlush 的完整执行过程。发送端 GPU 首先向 FIFO 写入同时包含 Data、Flag 和 Sync 语义的 Trigger，随后进入 fifo_.sync 等待。ProxyService 轮询到该 Trigger 后，先通过 conn.write 下发 Payload 的 RDMA 写，NIC 将数据直接搬运到对端显存；然后 ProxyService 调用 semaphore.signal，通过 RDMA 写更新对端 inboundToken；最后执行 conn.flush，并在相关请求处理完成后推进 FIFO 状态，使发送端 GPU 的同步等待得以返回。与此同时，对端 GPU 在 wait 中以 Acquire 语义观察 Token，在 Token 到达后即可安全读取对应数据。

因此，与上一节 MemoryChannel 的 GPU 直接远端访存模式相比，PortChannel 并不是让 CPU 重新承担数据拷贝，而是让 CPU Proxy 承担网络控制面的提交工作，将实际数据面交给 RDMA NIC。它在保留 GPU-driven 编程入口的同时，适配了跨节点网络所需要的队列管理、传输提交和完成语义，是 MSCCL++ 面向大消息和多机通信的重要通道。接下来，我们再看第三种 Channel——SwitchChannel。它不再依赖 GPU、CPU Proxy 和 NIC 之间的软件协同，而是进一步利用 NVLS 和 multimem 指令，将部分集合通信归约操作直接下沉到交换结构硬件中完成。

## 13. SwitchChannel

上一节介绍的 PortChannel，通过 GPU、CPU Proxy 和 RDMA NIC 的三级协作解决了跨节点数据传输问题。SwitchChannel 则采用了另一种优化思路：当系统具备 NVLS，也就是 NVLink SHARP 相关硬件能力时，不再由软件组织多轮点对点传输来完成归约，而是由 GPU 线程直接发出 multimem 指令，把跨 GPU 的数据聚合和分发操作下沉到 NVLS 交换结构中执行。因此，这一页关注的重点不是如何提交一次网络搬运，而是如何利用互联硬件本身提供的集合操作能力，缩短归约路径并减少软件调度开销。

SwitchChannel 的核心价值，可以概括为“一条 GPU 指令触发一次硬件级集合操作”。以 reduce 接口为例，GPU 线程执行 multimem.ld_reduce 指令后，NVLS 会访问参与 GPU 对应位置的数据，在交换结构内部完成指定的归约运算，并将归约结果直接返回给发起访问的 GPU。与传统 Ring AllReduce 需要经过多个通信轮次、逐跳传递数据相比，这种方式把原本由多个 send、recv 和 reduce 步骤组成的过程，压缩为一次面向 multimem 地址的硬件归约操作。

从接口层面来看，reduce、broadcast 和 multimemStoreReduce 分别对应三类典型能力。reduce 通过 multimem.ld_reduce 在读取数据的同时完成跨 GPU 归约；broadcast 通过 multimem.st 将一个值写入所有参与 GPU 对应的内存位置；multimemStoreReduce 则通过 multimem.red 在交换结构中完成归约，并将结果写回相关 GPU。也就是说，SwitchChannel 不只是提供数据搬运接口，而是直接暴露了带有集合语义的硬件操作，使归约和广播不再需要由软件拆解为多次普通内存访问。

页面中的结构图展示了这一过程。多个 GPU 分别针对同一个集合内存位置发起 multimem 操作，请求进入 NVLS Switch 后，由交换结构同步读取各个 GPU 的输入并执行归约。硬件得到最终结果后，再将结果返回到相应 GPU。这里最关键的变化，是数据不需要先集中到某个 GPU 上由 SM 执行归约，也不需要沿 Ring 拓扑经过多轮转发；交换结构本身就是归约执行单元，因此能够显著减少数据往返次数和中间状态。

时序图进一步强调了 SwitchChannel 的并行性。GPU0、GPU1 以及其他参与 GPU 可以面向同一个 multimem 地址发起归约指令，NVLS 在硬件内部聚合这些输入，并将结果返回给请求方。从 GPU Kernel 的视角来看，开发者使用的是一条 load-reduce 或 store-reduce 指令；但从硬件数据路径来看，这条指令实际表达的是一次跨多个 GPU 的集合操作。这也是 SwitchChannel 与 MemoryChannel、PortChannel 最本质的区别：前两者主要提供远端数据访问和传输能力，而 SwitchChannel 进一步把归约语义下沉到了互联硬件。

在数据类型和访存粒度方面，SwitchChannel 支持 f32、f16、bf16、FP8、i32、u32 和 f64 等多种类型，并能够使用 x1、x2、x4、x8 或 x16 等向量化形式进行操作。向量化能够让一条指令处理更多连续数据，但实际可用的数据类型、向量宽度和归约操作仍然受到具体 GPU 架构与 NVLS 指令能力的约束。因此，在设计 Kernel 时，需要同时考虑数据布局、地址对齐以及目标硬件能够支持的 multimem 指令组合。

需要注意的是，SwitchChannel 对内存有更严格的要求。mcPtr 必须指向 multimem-addressable 内存，使多个 GPU 能够通过统一的集合地址访问各自对应的数据；devicePtr 则通常用于保存本地结果或承接归约输出。这意味着普通的独立显存指针不能直接替代 multimem 内存，运行时需要完成特定的内存分配、映射和注册，并保证参与 GPU 之间满足相应的对称地址或特殊内存布局要求。

另外，页面中列出的 multimem 指令采用 relaxed.sys 语义。Relaxed 只保证指令本身的原子性或操作语义，并不会自动为前后的普通内存访问建立完整的顺序关系。因此，当归约结果依赖此前写入的数据，或者后续计算必须等待归约结果稳定可见时，Kernel 仍然需要根据生产者—消费者关系插入合适的 fence、barrier 或其他同步操作。硬件完成归约，并不等同于程序天然获得了完整的 Release/Acquire 顺序保证。

最后，SwitchChannel 是一条与硬件能力强相关的高性能路径。它依赖 NVLS、NVLink SHARP 以及可被 multimem 访问的内存资源，并不是所有 GPU 型号和系统拓扑都能够使用。当运行环境不具备这些条件时，MSCCL++ 需要回退到 MemoryChannel、PortChannel，或者由 NCCL 等通用通信路径完成相同的集合操作。因此，SwitchChannel 的意义并不是替代所有其他 Channel，而是在硬件条件满足时，为归约和广播提供一条更短、更直接的软件到硬件映射路径。

通过这三种 Channel 可以看到，MSCCL++ 会根据通信范围和硬件能力选择不同的数据路径：MemoryChannel 面向节点内直接访存，PortChannel 面向跨节点 RDMA 传输，SwitchChannel 面向 NVLS 硬件集合操作。理解这些通道之后，下一步还需要继续关注它们共同依赖的内存基础，也就是 RegisteredMemory、远端访问元数据以及对称 Offset 等内存布局约束。

## 15. RegisteredMemory 与对称 Offset

前面三种 Channel 虽然采用了不同的数据路径，但它们都有一个共同前提：通信端不能只拿到一段普通内存的本地指针，还必须知道这段内存是否能够被相应的 Transport 访问，以及远端应该通过什么方式定位和操作它。因此，这一页从 Channel 的执行机制进一步下沉到内存基础，重点介绍 RegisteredMemory 和对称 Offset。可以先给出一个核心结论：MSCCL++ 中的 RegisteredMemory 不是单纯的地址封装，而是一份同时包含本地内存范围和远端访问元数据的通信资源描述。

对于本地程序来说，一个裸指针通常只能说明虚拟地址从哪里开始，却不能直接支持跨进程、跨设备或者跨节点访问。不同 Transport 需要不同类型的附加信息。例如，CudaIpc 路径需要能够在另一个进程中重新打开显存映射的 IPC Handle；InfiniBand 或 RoCE 路径需要完成 Memory Region 注册，并生成网卡执行 RDMA 所需的 local key、remote key 以及地址范围；NVLS 路径则要求内存具备 multimem-addressable 属性。RegisteredMemory 的作用，就是把这些与 Transport 相关的信息和原始 Buffer 关联起来，使 Connection 和 Channel 能够基于统一对象构造远端访问能力。

从通信建立流程来看，发送端和接收端会先分别注册本地 Buffer，再通过控制面交换内存描述符。交换的不是实际 Payload，而是远端访问这段内存所需要的 Handle、Key、基地址、长度以及其他 Transport Metadata。连接建立后，Channel 会把这些信息进一步整理为 GPU Kernel 或 ProxyService 可以使用的 Device Handle。这样，在真正执行 put、get、RDMA Write 或 multimem 操作时，数据面不需要再次经过 Host 中转，也不需要临时查询远端内存信息，而是可以直接根据预先建立的访问描述完成传输。

这一页还强调了对称 Offset 的概念。所谓对称，并不一定要求不同 Rank 上的 Buffer 具有完全相同的虚拟地址，而是要求同一逻辑数据块在各 Rank 的注册内存区域中具有一致的相对偏移。例如，chunk1 在 Rank0、Rank1 和 Rank2 的 Buffer 中都位于相对于各自基地址的同一个 Offset。这样，通信算法只需要描述“访问第几个 Chunk”或者“从哪个 Offset 开始”，运行时就可以使用远端基地址加统一 Offset 的方式定位对应数据，而不必为每个 Rank 维护一套完全不同的地址计算逻辑。

图中的多个 Rank 都按照相同顺序组织 chunk0、chunk1、chunk2 和 chunk3，其中同名 Chunk 处于相同的相对位置。假设算法需要访问 chunk1，那么对于任意目标 Rank，都可以复用同一个 Offset，只需要替换该 Rank 对应的 RegisteredMemory 基址和访问元数据。这种布局能够显著简化集合通信算法中的地址生成，也便于 DSL、Executor 和 Primitive 层把逻辑 Chunk 映射到具体的远端内存位置。对于 AllGather、ReduceScatter 等规则分片的集合通信，对称布局尤其重要。

对称 Offset 还能够降低设备端控制信息的体积。如果每次通信都必须传递完整的远端绝对地址，Channel Handle 和 Trigger 中就需要携带更多针对单个 Rank 的地址数据；采用“RegisteredMemory 基址加 Offset”的方式之后，算法只需要传递较小的偏移和长度信息，远端地址可以由既有元数据快速计算。这也解释了为什么 PortChannel 的 Trigger 可以重点描述 Offset 和 Size，而不需要在每次操作中重新完成完整的内存注册与地址交换。

不过，zero-copy 并不意味着没有成本，而是把一部分运行时拷贝成本转化成了更严格的初始化和布局约束。应用需要提前确定 Buffer 的生命周期和大小，完成注册与元数据交换，并在通信期间保证内存不会被释放、重新分配或越界访问。对于需要对称布局的算法，各 Rank 还必须对 Chunk 顺序、Offset、对齐方式和容量形成一致约定。只要其中一个 Rank 的布局不同，相同 Offset 就可能指向错误数据，甚至造成非法远端访问。

因此，RegisteredMemory 同时承担了性能和安全边界两方面的职责。从性能角度看，它让 GPU、NIC 或 NVLS 能够直接访问预注册的用户 Buffer，减少中间拷贝和重复控制面操作；从正确性角度看，它限定了远端可以访问的内存范围，并提供 Transport 验证访问权限所需的 Handle 或 Key。实际使用时，还需要把 RegisteredMemory 的生命周期与 Connection、Channel 以及正在执行的 Kernel 或 RDMA 请求协调起来，确保所有异步通信完成之后才能注销或释放对应内存。

总结来看，MSCCL++ 的 zero-copy 能力并不是简单地绕过一次 memcpy，而是建立在“内存预注册、元数据预交换、地址按 Offset 解析、布局在各 Rank 间保持一致”这一整套机制之上。RegisteredMemory 解决远端能否访问的问题，对称 Offset 解决算法如何统一定位数据的问题，两者共同构成 MemoryChannel、PortChannel 和 SwitchChannel 的内存基础。在明确了这些底层资源之后，下一节就可以继续向上看 DSL 如何通过 Rank、Buffer、Chunk 和 Channel 等对象，把具体的内存位置和通信动作组织成可编译的集合通信算法。
